PARTE I: GIT (Filosofía y Línea de Comandos)

1. ¿Qué es GIT?

**Git** es el sistema de control de versiones **más utilizado** en el mundo. Es un **Sistema de Control de Versiones Distribuido (DVCS)**, lo que significa que en lugar de depender de un único servidor central, cada desarrollador tiene una copia completa del historial del proyecto, asegurando redundancia y facilitando el trabajo sin conexión. Fue diseñado para la **eficiencia y la integridad** de los datos.

Metáfora:

Piensa en Git como una máquina del tiempo para tu código, capaz de retroceder o avanzar a cualquier punto exacto de la historia del desarrollo.

2. Primeros Pasos y Área de Preparación

1. Directorio de Trabajo

Donde editas los archivos. Tu sandbox actual.

2. Área de Preparación (Staging)

Los cambios que has **marcado** para el próximo commit. El puente.

3. Repositorio Local

Donde se almacenan los commits permanentes.

El **Área de Preparación** es fundamental. Es donde decides *exactamente* qué cambios van a ser incluidos en la próxima "instantánea" de tu proyecto. El comando clave para usarla es:

# Agrega 'archivo.js' al Staging
git add archivo.js
# O agrega todos los cambios de archivos rastreados
git add .

3. Commits y Conventional Commits

Los Commits: La Instantánea Guardada

Un **Commit** es el registro formal de un cambio en el historial. Debe ser **atómico** (hacer solo una cosa) y tener un mensaje claro que describa lo que se hizo.

Conventional Commits: Estandarizando el Mensaje

Es una convención que exige que los mensajes de commit sigan una estructura estricta, lo cual ayuda a la legibilidad y a la automatización de herramientas:

feat: (Feature)

Introduce una **nueva funcionalidad** al código. Ej: `feat: Añadir formulario de contacto`

fix: (Fix)

Corrige un **error** (bug) de la aplicación. Ej: `fix: Corregir cálculo de impuestos en carrito`

chore: (Tarea)

Cambios que no tocan código de producción (ej. configuración, scripts de build).

refactor: (Refactorizar)

Cambio en la estructura del código sin cambiar su funcionalidad externa.

4. Ramas, Conflictos y .gitignore

¿Qué son las Ramas (Branches)?

Las ramas son una herramienta de aislamiento: te permiten trabajar en nuevas ideas o correcciones **sin afectar** la versión principal estable. La rama por defecto suele ser main.

# 1. Crear y moverse a una nueva rama
git checkout -b feature/nuevo-menu
# 2. Volver a la rama principal para fusionar
git checkout main
git merge feature/nuevo-menu

Ejemplo de Flujo de Ramas (Feature Branching)

Este es el flujo más común en desarrollo colaborativo. Aísla la nueva funcionalidad (feature/X) de la rama principal (main) y solo se fusiona tras una revisión exitosa (Pull Request).

Paso 1: Se parte de la rama estable.
    main (Commit A) --- (Commit B)
Paso 2: Se crea una nueva rama para la funcionalidad (feature).
    main (Commit B)
    └── feature/carrito-v2 (Commit B) <- Trabajando aquí
Paso 3: Se hacen commits en la feature.
    main (Commit B)
    └── feature/carrito-v2 (Commit C) --- (Commit D) <- Trabajo aislado
Paso 4: La feature se fusiona (Merge) de nuevo a la rama principal (main).
    main (Commit B) ------------------ (Commit E: Merge)
    └── feature/carrito-v2 (Commit C) --- (Commit D) <- Se elimina la rama después del merge

Conflictos en Git (Merge Conflicts)

Ocurren cuando Git no puede decidir qué cambio mantener automáticamente (ej. dos personas modificaron la misma línea de código). Debes resolverlo manualmente editando el archivo.

Ejemplo de Marcadores de Conflicto:

<<<<<< HEAD # Versión en mi rama actual
   console.log("Versión de desarrollo");
======== # Separador de Git
   console.log("Versión estable y final");
>>>>>>> main-branch # Versión de la rama que quiero traer

El Poder del .gitignore

El archivo **.gitignore** es crucial. Indica a Git qué archivos o directorios debe **ignorar y no rastrear**. Esto evita subir al repositorio archivos innecesarios como módulos de dependencia (`node_modules`), datos temporales o claves secretas (`.env`).

# Archivos generados automáticamente o privados
/node_modules
*.log
.env
/dist

5. Comandos Esenciales de Git (El Kit de Supervivencia)

git status

Muestra el estado de los archivos en el Directorio de Trabajo y Staging.

git add .

Añade **todos** los archivos modificados o nuevos al Área de Preparación (Staging).

git commit -m "..."

Guarda los cambios preparados en el historial del repositorio local.

git push

Sube los commits locales a la versión remota del repositorio (ej. GitHub).

git pull

Descarga y fusiona los cambios del repositorio remoto en tu rama local.

git log

Muestra el historial de commits de la rama actual.

git reset --soft [commit]

Deshace commits manteniendo los cambios en Staging.

PARTE II: GITKRAKEN (La Herramienta Visual)

¿Qué es GitKraken? El GUI Definitivo

**GitKraken** es un cliente Git de escritorio con interfaz gráfica de usuario (GUI) diseñado para simplificar tareas complejas de Git y hacer que el flujo de trabajo sea más rápido e intuitivo. Transforma comandos abstractos en acciones visuales.

  • Elimina la necesidad de memorizar comandos.
  • Mejora la comprensión del historial del proyecto.
  • Acelera la resolución de conflictos.

Todas sus Funcionalidades Clave

1. El Gráfico Visual (The Graph)

Muestra el **historial de commits** como un árbol de decisiones visual. Puedes ver en un vistazo dónde nacen las ramas, dónde se fusionan (Merge) y dónde ocurre el **Rebase**, haciendo comprensible la topología del código.

2. Staging por Líneas y Archivos

Permite arrastrar archivos al área de preparación. Su característica más potente es el **"Hunk Staging"**, que te deja seleccionar y añadir solo líneas específicas dentro de un archivo modificado, sin usar la terminal.

3. Operaciones Drag-and-Drop (Arrastrar y Soltar)

Simplifica las acciones de Merge, Rebase y Squash (combinar commits). Simplemente arrastras una rama sobre otra y GitKraken te presenta las opciones disponibles para fusionarlas.

4. Editor de Conflictos Integrado

Ofrece una vista de **tres vías** (tu versión, la versión entrante, y el resultado final) que te permite hacer clic para seleccionar qué bloque de código mantener, haciendo la resolución de conflictos mucho más rápida y menos propensa a errores que en la terminal.

5. Botones de Undo y Reflog Visual

Incluye botones directos de **"Undo"** para revertir commits, lo que mitiga el miedo a cometer errores. Además, visualiza el Reflog (registro de referencias de Git) permitiendo recuperar trabajo que pudo haberse "perdido" fácilmente.

6. Integración con Plataformas

Se integra nativamente con GitHub, GitLab, Bitbucket y Azure DevOps. Permite crear, ver, y gestionar las **Pull Requests** directamente dentro de la aplicación, optimizando el flujo de trabajo de revisión de código.
